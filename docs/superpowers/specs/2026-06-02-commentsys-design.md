# 扫码写评价系统 · 产品设计文档

> 日期:2026-06-02
> 适用场景:单店头皮管理 / 养发门店
> 一句话定位:**让顾客在店里 3 分钟内,写出一条高质量公域评价的店内辅助工具**

---

## 1. 项目背景与目标

### 1.1 背景
头皮管理 / 养发是典型「高客单 + 决策长 + 重口碑」品类。顾客单次客单价 200-800 元,90% 决策依据是大众点评、抖音的店铺评分和评价。但顾客在店里写评价的转化率天然很低,因为:
- 服务结束后顾客直接走,没有"写评价"的钩子
- 想写也不知道写什么,凑半天 50 字
- 拍照费劲,没好图
- 一条文案不知道发哪个平台合适

### 1.2 目标
v1 上线后 3 个月达到:
- 月均扫码评价填写量 ≥ 30 次
- 公域评价(主要点评)月新增 ≥ 20 条
- 顾客平均完成时间 ≤ 3 分钟
- 1-3 星差评 100% 拦截在私域,公域不出现差评

### 1.3 非目标(v1 明确不做)
- 不做拓客 / 加粉 / 引流(那是另一套系统)
- 不做会员体系 / 积分 / 优惠券
- 不取代点评、抖音、小红书,只做"往里送水"
- 不做多门店管理

---

## 2. 核心场景与角色

| 角色 | 主要操作 | 终端 |
|---|---|---|
| **顾客** | 扫桌贴码 → 打分 → (好评)选平台/标签 → 一键复制跳转 / (差评)私域留言 | 手机 H5 |
| **店主** | 看数据看板 / 处理差评 / 管照片库 / 改技师项目 | 手机 H5 后台 |
| **店长** | 处理差评(主) / 看数据看板 | 手机 H5 后台 |

**关键决策:店主和店长各自手机号登录,v1 不区分权限,但操作日志留痕。**

## 5. AI 文案生成策略

### 5.1 模型选择

**首选 DeepSeek-V3**(国产、便宜、中文质量好),备选通义 qwen-turbo。

| 维度 | DeepSeek-V3 | 通义 qwen-turbo |
|---|---|---|
| 输入价格 | 约 0.5 元 / 百万 tokens | 约 0.3 元 / 百万 tokens |
| 输出价格 | 约 2 元 / 百万 tokens | 约 0.6 元 / 百万 tokens |
| 中文质量 | ★★★★★ | ★★★★ |
| 服务稳定性 | 高 | 高 |

**成本估算:**
- 单次评价生成约 500-1000 tokens 输出
- 单条评价成本 < 0.002 元
- 一年 1 万条评价 ≈ 20 元

### 5.2 4 平台 4 套 Prompt

每个平台一套独立的 system prompt,核心差异:

| 平台 | 文案风格 | 长度 | 关键要素 |
|---|---|---|---|
| 大众点评 | 流水账、专业感、关键词密集 | 200-300 字 | 项目 + 技师 + 头皮变化 + 价格感受 + 推荐人群 |
| 美团 | 同点评,稍口语化 | 180-250 字 | 同上 |
| 抖音 | 口语化、短句、结果先行 | 80-150 字 | 一句话效果 + 服务流程 + 价格锚点 |
| 小红书 | 种草、emoji、分点列表 | 150-250 字 | "姐妹们" + 痛点共鸣 + 项目细节 + 出乎意料效果 |

**Prompt 模板示例(点评):**

```
你是一位刚做完头皮养发服务的真实顾客,在大众点评写一条好评。

要求:
- 字数 200-300 字,流水账风格,自然真实
- 必须包含项目名: {项目标签}
- 必须提到技师: {技师名}
- 评分:{星级}星
- 体验时段: {随机时段:上午/下午/晚上}
- 适合人群: {随机痛点:脱发严重的人/油性头皮/产后掉发/中年男性}
- 句式参考: 句式编号 {1-10 随机}
- 严格避免用词: 极致、震撼、完美、惊艳(平台风控敏感词)
- 不要使用 emoji
- 不要分段,自然流畅一段话
```

### 5.3 防查重

平台都有"评价内容查重"风控,完全重复的文案会被压权重甚至删除。

策略:
- 每次都重新调用 LLM,**不缓存复用**
- prompt 注入随机变量:技师 × 项目 × 时段 × 痛点 × 句式编号
- 控制句子长度抖动 ±20%
- 偶尔在 prompt 里加"轻微抱怨"指令(例如「可以提一下停车不方便」),让评价更真实

### 5.4 兜底:LLM 调用失败时回退到模板库

- 超时 / 限流 / 失败 → 自动切换到模板库随机组合(见 §6)
- 顾客无感,界面文案不变
- 后台记录 LLM 失败率,> 5% 报警

## 6. 模板库与变量池

### 6.1 设计理念

不是"几条固定模板",是 **"句式骨架 + 变量池组合"**。

```
模板骨架(点评示例):
{开场} {项目体验} {技师评价} {效果描述} {价格感受} {结尾推荐}

变量池(单店首批种子量):
- 开场池:20 条
- 项目体验池:8 条(对应 8 个项目)
- 技师评价池:15 条
- 效果描述池:20 条
- 价格感受池:10 条
- 结尾推荐池:10 条

理论组合数:20 × 8 × 15 × 20 × 10 × 10 = 480 万种
实际不会全枚举,每次随机抽取,做去重(同一句式骨架 24 小时内不复用)
```

### 6.2 4 平台分别建变量池

每个平台单独一套变量池(点评的"开场"和小红书的"开场"完全不同的调性)。

### 6.3 内容来源

- 第一版:**店主自己写 50 条种子文案**,我们(开发方)切片成变量池
- 上线后:从顾客实际编辑过的评价中,人工筛选优质内容反哺变量池
- v2 可加管理后台编辑器,v1 暂用配置文件

## 7. 照片库管理

### 7.1 内容要求

**严禁用网图。** 必须真实店内拍摄,平台风控会查图片哈希。

第一版需要店主自拍 **30-50 张**,分三类:

| 类型 | 数量 | 拍摄内容 |
|---|---|---|
| 环境图 | 10-15 | 招牌、前台、休息区、护理间(有/无人各几张) |
| 过程图 | 15-20 | 头皮检测仪屏幕、洗发、按摩、敷膜、吹风(避免拍到客人脸) |
| 效果图 | 10-15 | 头皮镜检前后对比、产品瓶、技师工作中 |

### 7.2 元数据(每张图必填)

```json
{
  "id": "photo-001",
  "url": "https://cdn.../001.webp",
  "type": "环境 | 过程 | 效果",
  "platforms": ["点评", "美团", "抖音", "小红书"],
  "rating_match": [4, 5],  // 适合的星级
  "tags": ["头皮检测", "防脱护理"],
  "uploaded_at": "2026-06-02T10:00:00Z"
}
```

### 7.3 顾客端展示逻辑

- 根据顾客选的「平台 × 项目标签 × 星级」过滤
- 从匹配池中随机抽 5 张展示
- 顾客勾选 2-3 张
- 同一张图同一顾客只能选一次

### 7.4 图片管理(后台)

- 店主 / 店长在管理 H5 上传图片
- 上传时填元数据(默认值 + 手动调整)
- 自动压缩到 < 500KB(webp 格式)
- 支持删除、改标签

- 店主、店长各自手机号 + 短信验证码
- 同一手机号下次免登录(7 天 cookie)
- 操作日志记录"谁登录、谁操作了什么"

### 8.2 首页 · 数据看板

```
┌────────────────────────────────┐
│  📅 本周                         │
│                                  │
│  扫码次数 142                    │
│  ⭐⭐⭐⭐⭐ 108  ⭐⭐⭐⭐ 22       │
│  ⭐⭐⭐ 7    ⭐⭐ 4    ⭐ 1     │
│  公域跳转 119                    │
│   点评 78 · 美团 12              │
│   抖音 23 · 小红书 6             │
│  差评待处理 🔴 2                 │
│                                  │
│ [今日][本周][本月]               │
```

**关键限制(必须告诉店主):**
- "公域跳转 119" 只统计到"点击复制+跳转 App"这一步
- 顾客**是否真的发布**了评价,平台不会回传,系统无法知道
- 店主需要自己去点评/抖音店铺后台核对实际新增评价数

### 8.3 差评中心(核心功能)

每条 1-3 星留言一张卡片,按时间倒序:

```
┌────────────────────────────────┐
│ 🔴 2 星  10 分钟前              │
│ "技师按摩力度太轻,效果不明显"│
│ 联系方式:138****5678          │
│ 提交位置:店内 / 设备:iPhone   │
│                                  │
│ [📞 打电话] [✅ 标记已处理]      │
│ [📝 添加跟进记录]                │
└────────────────────────────────┘
```

**通知联动:**
- 提交时立刻推送企业微信群机器人(店主 + 店长共群)
- 群消息内容包含星级、留言、联系方式、跳转后台链接
- 5 分钟内无人点"已处理",再推一次

### 8.4 设置 · 极简配置

只 4 项:

| 配置项 | 改动频次 |
|---|---|
| 门店基础信息(名称、电话、地址) | 几乎不变 |
| 技师名单(增删改) | 月度变动 |
| 项目名单 | 季度变动 |
| 4 平台店铺链接(用于跳转) | 一次配置 |

更高级的配置(模板池、AI prompt、星级阈值)**v1 不开放**,由开发方维护。

## 9. 差评通知与处理

### 9.1 触达方式:企业微信群机器人

**前置准备(店主一次性):**
- 注册企业微信(免费,5 分钟)
- 建一个内部群(店主 + 店长 2 人即可)
- 群里添加"群机器人",拿到 webhook URL
- 在系统后台填入 webhook URL

**推送内容示例:**

```
🔴 新差评待处理

评分: ⭐⭐ 2 星
时间: 2026-06-02 14:30
留言: 技师按摩力度太轻,效果不明显
联系: 138****5678

[点击进入后台处理]
```

### 9.2 处理 SLA

- 收到通知 5 分钟内电话联系顾客
- 顾客同意补救(免费补做、升级护理、退款) → 后台标记"已处理"
- 顾客拒绝补救 / 联系不上 → 仍标记"已处理",留备注

### 9.3 不做的事(v1)

- 不做"自动回复" — 差评必须人工介入
- 不做"差评导出报表" — 数据量小,直接看后台

## 10. 技术架构选型

### 10.1 整体架构(MVP)

```
┌────────────┐         ┌────────────────────┐
│ 顾客端 H5  │         │  店主端 H5         │
│ (Mobile)   │         │  (Mobile)          │
└──────┬─────┘         └────────┬───────────┘
       │                         │
       │      HTTPS / JSON       │
       └─────────────┬───────────┘
                     ↓
        ┌──────────────────────┐
        │   后端 API 服务       │
        │   (Node.js + Hono /   │
        │    Python + FastAPI)  │
        └──────┬───────────────┘
               │
   ┌───────────┼─────────────┬───────────────┐
   ↓           ↓             ↓               ↓
┌──────┐  ┌────────┐  ┌────────────┐  ┌──────────────┐
│ 数据库│  │ 对象存储│  │ DeepSeek/  │  │ 企业微信群    │
│PG/SQLite│ │(图片) │  │ 通义 LLM   │  │ 机器人 webhook│
└──────┘  └────────┘  └────────────┘  └──────────────┘
```

### 10.2 推荐技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 H5 | Vue 3 / React + TailwindCSS | 移动端 H5 体验好,Tailwind 上手快 |
| 路由 | 顾客端 Hash 路由 / 后台 History 路由 | 顾客端要在微信里跑,Hash 路由更稳 |
| 后端 | Node.js + Hono(或 Python + FastAPI) | 轻量、单店流量小、二选一看团队语言 |
| 数据库 | SQLite(MVP) → PostgreSQL(规模化) | 单店数据量小,SQLite 完全够,运维零成本 |
| 图片存储 | 阿里云 OSS / 七牛 / 腾讯 COS | 国内访问快,单店一年 < 50 元 |
| LLM | DeepSeek-V3 API | 中文好、便宜、稳定 |
| 通知 | 企业微信群机器人 webhook | 免费、实时 |
| 部署 | 阿里云 ECS 单机 / Vercel | 单店一台 2C4G 服务器够用 |

### 10.3 一些必须强调的限制

- **微信内打开 H5 拦截 URL Scheme**:无法直接拉起点评/抖音 App
  - 解法:H5 顶部一直挂提示「请点右上角 → 在浏览器打开」
  - 复制评价文案在微信内能成功,跳转 App 需要顾客在浏览器内完成
- **iOS 剪贴板权限**:Safari 15.4+ 支持 navigator.clipboard.writeText,前置版本要降级
- **图片大小**:超 1MB 在 3G 网络下加载慢,统一压到 < 500KB

## 13. 实施风险与对策

| 风险 | 概率 | 影响 | 对策 |
|---|---|---|---|
| 微信内打开 H5 跳转 App 失败 | 高 | 中 | 顶部常驻"右上角浏览器打开"提示 + 店员现场口头引导 |
| 平台对 AI 生成评价的查重和压权重 | 中 | 高 | 4 平台独立 prompt + 变量池随机 + 文案长度抖动 + 不缓存 |
| 顾客嫌 5 步流程长不愿意填 | 中 | 高 | 总步骤 ≤ 5,3 分钟内完成;现场赠送一杯护理茶钩子 |
| 差评通知漏看 | 低 | 高 | 双人企微群同时收 + 5 分钟无人处理重推 |
| 照片库枯竭 / 重复使用被风控 | 中 | 中 | 季度更新照片库,每张图记录使用次数,超 50 次冷却 |
| LLM API 限流 / 失败 | 低 | 中 | 模板库自动兜底,顾客无感 |
| 顾客打分时直接选 5 星走完流程,真实满意度被掩盖 | 高 | 中 | 在差评中心外,后台增加"5 星但跳转后未发布"数据,反推真实满意度 |

## 14. 待用户进一步明确的事项
